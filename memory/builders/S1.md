## What changed

**Before:** `gt-auth` cookie was `sha256("gt:" + APP_PASSWORD)`. Static value, same every session, never rotated, no way to revoke short of changing APP_PASSWORD itself. If anyone ever captured that cookie (XSS, shoulder-surf, network log, leaked screenshot), they owned the terminal for 30 days.

**After:** `gt-auth` cookie is a signed JWT (HS256) with `iat` + 30-day `exp`. Backed by `SESSION_SECRET` тАФ rotating that env var invalidates every outstanding session immediately. Drop-in cookie-name compatible so no frontend code changed.

## Bugs fixed along the way

### Bug 1: Sentry `withSentryConfig` breaks middleware detection when the file is at `src/middleware.ts`
Symptom: middleware manifest empty (`"middleware": {}`), Next.js never compiles middleware, all routes public. Requests to `/` return 200 instead of redirecting to `/login`. Dev server logs show every route compiling but never "Compiling /middleware".

Root cause: Sentry's `withSentryConfig` Webpack plugin (v10.49.0) only picks up middleware at the project root, not inside the `src/` directory layout. The pre-S1 middleware at `src/middleware.ts` had been working until Sentry landed one commit earlier (S2) тАФ nobody noticed because the E2E tests run against prod (where Sentry's prod bundling is different) not local dev.

Fix: moved `middleware.ts` to project root. Single line of pre-existing `src/middleware.ts` code, just different path. All other files that import from `@/lib/session` continue to work through the `@` alias.

### Bug 2: `APP_PASSWORD` in `.env.local` gets silently shell-interpolated
Symptom: Login with the "obvious" password `Glastonbury$2026#GT!` returns 401. Login with `Glastonbury#GT!` returns 200.

Root cause: Next.js's dotenv loader interpolates `$VAR` references inside `.env.local`. `$2026` is not a known var, so it expands to empty string. The effective APP_PASSWORD becomes `Glastonbury#GT!`. The E2E global-setup at `e2e/global-setup.ts` already works around this (tries multiple password variants).

**Action for Wes:** To fix, either:
- **(a)** Escape the dollar in `.env.local`: `APP_PASSWORD='Glastonbury$2026#GT!'` (single quotes around the value disable dotenv interpolation)
- **(b)** Leave it тАФ effective password is `Glastonbury#GT!`. Vercel env vars don't do shell interpolation so prod is unaffected if you want the real one there.

Not fixed in this commit тАФ requires editing `.env.local` which is in the agent deny list (correctly).

## One-time setup for Wes (in Vercel env)
1. Generate a fresh session secret: `openssl rand -base64 48`
2. Add `SESSION_SECRET = <that value>` to Vercel env (Production + Preview + Development)
3. Deploy тАФ every existing session becomes invalid instantly (intentional тАФ kills any stale cookie captured before S1 shipped). You log in once with `Glastonbury#GT!` and get a fresh JWT.
4. To rotate sessions in the future (e.g., suspected breach), just change `SESSION_SECRET` and redeploy. All tokens invalidate on next request.
