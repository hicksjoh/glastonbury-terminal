## Scope
80+ API routes and every page component error logged to `console.error` → Vercel logs → /dev/null when nobody's reading. Switching to Sentry gives: structured capture, correlation across server/edge/client, stacks symbolicated by release, alert rules (Slack/email/SMS).

Fire-and-forget: if `SENTRY_DSN` is not set, every SDK init is a safe no-op — the app ships without Sentry and no data leaves the machine.

## Files shipped
- `package.json`: `@sentry/nextjs@10.49.0` added (+171 transitive deps, 5 high-severity vuln in deps, all in dev/build toolchain — not in runtime path).
- `instrumentation.ts`: Next.js 14+ `register()` hook + `onRequestError()` for React Server Components and server actions.
- `sentry.server.config.ts`: Node-runtime init. Production-only by default.
- `sentry.client.config.ts`: Browser init using `NEXT_PUBLIC_SENTRY_DSN`. Replay disabled (avoid bundle bloat).
- `sentry.edge.config.ts`: Edge-runtime init for middleware + edge handlers.
- `pages/_error.tsx`: Pages-Router stub — Sentry's webpack plugin demands it even on App Router projects. Delegates to `Sentry.captureUnderscoreErrorException`.
- `next.config.js`:
  - Wrapped in `withSentryConfig(...)` with `silent`, `hideSourceMaps`, `widenClientFileUpload`, `tunnelRoute: '/monitoring'` (so ad-blockers don't swallow client errors).
  - CSP `connect-src` updated with `https://*.ingest.sentry.io https://*.sentry.io`.

## Bug fixes landed in this commit (build surfaces these; Sentry wrapper triggers the strict check)
Next.js's `next build` typecheck is stricter than `tsc --noEmit` because it pulls in auto-generated `.next/types/**`. Fixed 4 pre-existing TS errors that blocked the build:
- `src/app/earnings/live/[sessionId]/page.tsx:52` — `params.sessionId` → `params?.sessionId ?? ''`
- `src/app/research/[id]/page.tsx:30+47` — same pattern, extracted `memoId` variable
- `src/app/stock/[symbol]/page.tsx:49` — `params.symbol` → `params?.symbol ?? ''`
- `src/app/trading/page.tsx:79` — `searchParams.get(...)` → `searchParams?.get(...)`

These had always been broken under strict-null-checks but only got caught when I forced a fresh production build. Sentry didn't create these — it just exposed them.

## One-time setup for Wes (blocks full observability, not deploy)
1. Sign up at https://sentry.io (free tier: 5K errors/month, 10K transactions)
2. Create a new project, type: "Next.js"
3. Copy the DSN (looks like `https://<hash>@<org>.ingest.sentry.io/<project>`)
4. Add to Vercel env (Production + Preview + Development):
   - `SENTRY_DSN` = the DSN from step 3
   - `NEXT_PUBLIC_SENTRY_DSN` = same DSN (exposed to browser intentionally — DSNs are not secrets)
   - `SENTRY_ORG` = your org slug (e.g., `glastonbury-group`)
   - `SENTRY_PROJECT` = project slug (e.g., `glastonbury-terminal`)
5. Optional (enables source-map upload for symbolicated stacks):
   - Create a Sentry auth token at https://sentry.io/settings/account/api/auth-tokens/ with `project:releases` + `org:read` scopes
   - Add `SENTRY_AUTH_TOKEN` = that token to Vercel env (Production only)
6. Configure alert routes in Sentry UI → email to hicksjoh@gmail.com on error spike, optional Slack channel
